## 0. 設計原則(不可違背)

1. **事實與規則分離**:出席/請假/遞補/繳費是「事實事件」,退費金額是「規則套用事實的推導結果」。資料庫永不儲存計算後的金額(期末結算快照除外)。
2. **隊員端零學習成本**:LINE 登入、單頁操作、每個動作 ≤ 3 次點擊。
3. **管理員端單一真相來源**:所有帳務數字可追溯到原始事件,不允許手動覆寫金額,只允許新增「調整事件」。
4. **規則可中途修改**:季中改退費規則時,歷史推導結果自動跟著重算(因為金額本來就不落地)。
5. **遞補配對全自動且確定性**:同場多人請假時,遞補額度依「請假時間先後」自動配對,不引入人工裁量。

## 1. 技術棧

| 層 | 選型 | 說明 |
|---|---|---|
| 前端 | Next.js 14+ (App Router) + TypeScript | 部署於 Vercel Hobby(免費) |
| UI | Tailwind CSS + shadcn/ui | 行動優先,隊員端以手機為主要視口 |
| 後端 / DB | Supabase Free (PostgreSQL) | Auth、RLS、RPC function 全部用 Supabase |
| 登入 | LINE Login (OAuth) via Supabase Auth 自訂 provider | 隊員與遞補者皆無密碼;管理員同樣走 LINE 登入後以角色區分 |
| 退費計算 | PostgreSQL View / RPC | 計算邏輯放 DB 層,前端只顯示 |

**成本**:全免費。唯一可選支出為自訂網域。Keep-alive 與備份排程(見第 6 節)為免費方案必要配套,不可省略。

**明確不做**:LIFF 包殼(v2 再評估)、LINE 推播通知(v2)、金流串接(繳費仍為線下轉帳,系統只記錄)。

## 2. 角色與權限

| 角色 | 權限 |
|---|---|
| 管理員 (admin) | 全部功能:季/場次/成員管理、代登事件、遞補管理、結算 |
| 隊員 (member) | 查看本季場次、自己的請假/出席/遞補狀態、退費試算;對未截止場次自助請假/取消請假 |
| 遞補者 (substitute) | LINE 登入後可瀏覽「開放遞補」的場次並報名/取消報名;查看自己的遞補紀錄與單場費 |

- 權限以 Supabase RLS 實作,規則寫在 migration 內。一個系統可有多名 admin。
- 同一人可同時是某季的隊員、另一季的遞補者(角色綁在季層級,不綁帳號)。

## 3. 資料模型

### 3.1 ER 概觀

```
seasons 1 ──< sessions(場次,含容量)
seasons 1 ──< season_members(季成員,含季費)
seasons 1 ──< season_rules(規則,版本化)
sessions 1 ──< events(事件:請假/取消請假/出席補登)
sessions 1 ──< session_subs(遞補報名,配對至請假事件)
season_members 1 ──< payment_events(繳費/調整事件)
profiles(LINE 使用者)1 ──< season_members
profiles 1 ──< session_subs
```

### 3.2 資料表定義

實作見 `supabase/migrations/0001_schema.sql`。重點欄位:

- `season_rules`:`season_fee`、`refund_per_session`、`sub_fee_per_session`(預設 225)、
  `leave_deadline_hours`(預設 2)、`refund_requires_sub`(預設 true)、`effective_from`(版本化)
- `sessions.capacity` 預設 18
- `events.type`:`leave` | `leave_cancel` | `attend_correction`(只追加)
- `session_subs.status`:`signed_up` | `withdrawn` | `no_show`
- `payment_events`:`season_member_id` 與 `session_sub_id` 二擇一;金額正 = 收款,負 = 退款/折讓
- `settlements`:結算快照,季結束時一次性寫入

註:v1.0 的 `max_refundable_leaves`(退費上限)已移除,確認本隊不設上限。

### 3.3 關鍵推導邏輯(DB View / RPC)

**A. 某成員某場次的請假狀態** = 該 (session, member) 最新一筆 event:
- 最新為 `leave` 且 `created_at` 早於(場次開始時間 − 2 小時)→ **有效請假**
- 最新為 `leave` 但晚於截止 → **逾期請假**(記錄但永不退費)
- 最新為 `leave_cancel` 或無事件 → **出席**

**B. 遞補配對(每場次獨立計算)**:
```
有效遞補數 m = count(session_subs WHERE status = 'signed_up')
有效請假者依 leave 事件 created_at 升冪排序,
前 m 名 → 「已遞補」(可退費)
其餘   → 「待遞補」(暫不可退費;若後續有人報名遞補則自動遞進)
```
- 遞補者撤銷報名(withdrawn)即釋出額度,配對即時重算。
- 遞補報名數不得超過該場「有效請假數」(無缺額時報名按鈕關閉)。
- 場次 `cancelled` 時:**全員視同有效請假且不受遞補條件限制**,一律退費;該場所有遞補報名自動作廢、已收單場費轉為應退。

**C. 退費試算** `fn_refund_preview(season_member_id)`:
```
可退場次 = count(狀態 = 已遞補 的有效請假) + count(被取消場次)
退費金額 = 可退場次 × refund_per_session   -- 取「生效中」規則
餘額     = 已繳總額 − (季費 − 退費金額)
```

**D. 遞補者帳務** `fn_sub_balance(profile_id, season_id)`:
應繳 = 有效遞補場次 × sub_fee_per_session;對照 payment_events 顯示已繳/未繳。

## 4. 頁面與流程

### 4.1 隊員端(行動優先)

**`/` 本季首頁(登入後唯一主頁)**
- 頂部:我的退費試算卡(已遞補 X 場 × $Z = 預估退費 $Y;另顯示「待遞補 N 場」)
- 場次列表(依日期排序,過去場次折疊):每列顯示日期/場地/我的狀態(出席、已遞補、待遞補、逾期請假)
- 未截止場次右側一顆按鈕:`請假` ↔ `取消請假`(toggle,confirm dialog 一次)
- 已截止場次按鈕 disabled 並顯示「已截止」
- 請假成功後 toast 提示:「退費將於有人遞補後生效,目前狀態:待遞補」

> 操作路徑驗收:開啟連結 → LINE 授權(首次)→ 點請假 → 確認。≤ 3 步。

**`/me` 我的紀錄**
- 本季請假明細(含 已遞補/待遞補/逾期 標記)、繳費紀錄、目前餘額

### 4.2 遞補端

**`/sub` 遞補場次列表**(LINE 登入後可見)
- 僅列出「有缺額」(有效請假數 > 有效遞補數)且未開打的場次
- 每列顯示:日期/場地/剩餘缺額/單場費
- 一鍵 `報名遞補` ↔ `取消報名`(開打前皆可取消)
- 我的遞補紀錄與應繳單場費

> 遞補連結可直接貼到 LINE 群組,任何人點開登入即可報名,先到先得。

### 4.3 管理員端

**`/admin/seasons`**:季列表、建立季(精靈式:基本資料 → 規則 → 批次產生場次,支援「每週 X 固定時段」自動展開,容量預設 18)

**`/admin/seasons/[id]`**:
- 場次管理:新增/取消場次、查看單場「請假 ↔ 遞補配對表」(顯示配對順序)、代登遞補(幫沒用系統的臨打報名)
- 成員管理:從 profiles 加入成員、標記退出、代登請假(代登事件記錄 created_by)
- 規則管理:顯示規則版本歷史,修改 = 新增版本
- 帳務總表(隊員):每位成員「應收 / 已繳 / 已遞補請假 / 待遞補請假 / 預估退費 / 餘額」即時表格
- 帳務總表(遞補者):每人「遞補場次 / 應繳 / 已繳」

**`/admin/seasons/[id]/settle`**:結算頁
- 預覽全員結算金額(含當下仍「待遞補」者列為不退費)→ 確認後寫入 settlements、季狀態改 'settled'、鎖定所有寫入
- 匯出 CSV(成員、退費金額、餘額;遞補者收費明細)

## 5. 驗收標準(關鍵案例)

1. 隊員於截止前(賽前 2 小時)請假後又取消,帳務試算回到 0 退費,缺額同步消失。
2. 某場 3 人有效請假、僅 2 人報名遞補:依請假時間最早的 2 人顯示「已遞補(退費)」,第 3 人「待遞補(不退費)」;第 3 位遞補者報名後,第 3 人自動轉為已遞補。
3. 遞補者撤銷報名,原本「已遞補」的最晚請假者自動退回「待遞補」,試算同步更新。
4. 季中將 `refund_per_session` 從 150 改成 100,所有人的試算即時反映新規則(歷史事件不動)。
5. 管理員取消某場次,全員該場一律退費(不受遞補條件限制),該場遞補報名與單場費自動作廢。
6. 逾期請假(賽前 2 小時內)顯示於紀錄但永不退費,也不開放遞補名額。
7. 結算後任何寫入操作(請假、繳費、遞補報名)被 RLS 拒絕。
8. 隊員 A 無法看見隊員 B 的繳費金額;遞補者只能看見自己的帳務(RLS 驗證)。
9. 中途加入成員(joined_at 晚於季開始):季費按剩餘場次比例計算(`季費 × 剩餘場次 / 總場次`,無條件進位至十位)。
10. 遞補報名數達到該場有效請假數時,報名按鈕關閉並顯示「已滿」。
